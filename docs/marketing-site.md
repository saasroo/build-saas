# Build a SaaS Marketing Site with Saasify

A step-by-step guide to building a complete SaaS marketing website using the [Saasify](https://saasify.dev) open-source Hugo theme.

## What You'll Build

A fully configured marketing site with:
- Homepage with hero, features, testimonials, and CTA
- Pricing page with plan comparison and FAQ
- About/Company page with team, values, and stats
- Feature pages for each product capability
- Blog with categories, tags, and sidebar
- Documentation hub with sidebar navigation
- SEO-optimized, responsive, 90+ Lighthouse scores
- 21 built-in shortcodes for rapid page building

---

## Prerequisites

Before you start, make sure you have the following installed:

- **Hugo Extended** v0.80.0+ — [download from GitHub releases](https://github.com/gohugoio/hugo/releases). It **must** be the Extended edition (required for TailwindCSS processing). Verify with `hugo version` — the output must include "extended".
- **Node.js** v14+ and npm — required for TailwindCSS compilation. Verify with `node --version` and `npm --version`.
- **Git** — required for installing the theme as a git submodule. Verify with `git --version`.

## Step 1: Plan Your Site

Before writing a line of code, spend 15 minutes answering these questions. You'll need the answers when filling in the config.

**Product basics:**
- What is your product called?
- What does it do in one sentence?
- What are the 2–3 core features you want to highlight?

**Pricing:**
- How many tiers? (Free / Pro / Enterprise is the most common pattern)
- What are the prices?
- What are the key features of each tier?

**Brand:**
- What is your primary brand color? (Pick a hex — e.g. `#2563eb` for blue)
- Do you have a logo SVG ready?

**Pages:**
- Which pages do you need? At minimum: Home, Pricing, Company, Blog, Docs
- Do you need a Careers page?

**CTAs:**
- What is your primary call-to-action? ("Get Started Free", "Start Trial", etc.)
- Where does it go? (Your app sign-up URL)

Having clear answers now will let you fill in the config and content placeholders in one pass without stopping to think.

## Step 2: Install Hugo + Saasify

### Create a new site

Run these commands to scaffold a new Hugo site and install the Saasify theme:

```bash
# Create new Hugo site
hugo new site my-saas-site
cd my-saas-site

# Initialize git
git init

# Add the Saasify theme as a git submodule
git submodule add https://github.com/chaoming/hugo-saasify-theme.git themes/hugo-saasify-theme
git submodule update --init --recursive

# Copy build config files to the site root
cp themes/hugo-saasify-theme/package.json .
cp themes/hugo-saasify-theme/postcss.config.js .
cp themes/hugo-saasify-theme/tailwind.config.copy.js ./tailwind.config.js

# Install npm dependencies
npm install
```

The copy step is important — you need `package.json`, `postcss.config.js`, and `tailwind.config.js` at the **site root**, not just inside the theme directory.

### Adding to an existing site

If you already have a Hugo site:

```bash
# Add theme as submodule (from site root)
git submodule add https://github.com/chaoming/hugo-saasify-theme.git themes/hugo-saasify-theme
git submodule update --init --recursive

# Copy build config files
cp themes/hugo-saasify-theme/package.json .
cp themes/hugo-saasify-theme/postcss.config.js .
cp themes/hugo-saasify-theme/tailwind.config.copy.js ./tailwind.config.js

# Install npm dependencies
npm install
```

Then add `theme = "hugo-saasify-theme"`, `[build] writeStats = true`, and `[module.hugoVersion] extended = true, min = "0.80.0"` to your existing `hugo.toml`.

### Directory structure after install

```
my-saas-site/
├── archetypes/
├── assets/
├── content/             # Your site content (Markdown)
├── data/
├── i18n/                # Translation files (create if multilingual)
├── layouts/             # Override theme layouts here
├── node_modules/        # npm dependencies (gitignored)
├── static/
│   └── images/          # Your images
├── themes/
│   └── hugo-saasify-theme/  # The theme (git submodule)
├── hugo.toml            # Site configuration
├── package.json         # Copied from theme
├── package-lock.json    # Generated by npm install
├── postcss.config.js    # Copied from theme
└── tailwind.config.js   # Copied from theme (customizable)
```

### Running the dev server

```bash
# Development (with TailwindCSS hot reload)
npm start
# Site available at http://localhost:1313

# Production build
npm run build
# Output in public/ directory
```

**Important**: Always use `npm start` instead of `hugo server`. The npm script runs TailwindCSS compilation alongside Hugo. Running `hugo server` directly will result in missing styles.

**Key sections to fill in:**

- `[params]` — one-line description and company name
- `[params.cta]` — the headline and description for the CTA section that appears at the bottom of every page; update `gradient_from` and `gradient_to` to match your brand colors
- `[params.header.buttons]` — update the URLs to point to your sign-in and sign-up pages
- `[[menu.main]]` — customize the nav links and feature submenu items
- `[[menu.footer_column_*]]` — customize footer links to match your nav

The `[build] writeStats = true` setting is required for TailwindCSS to know which classes are used. Do not remove it.

Open `tailwind.config.js` in your site root (not the one inside the theme directory). This is where you define your primary and secondary brand colors.

### How the shade scale works

The theme uses shades 50–900. The most important one is **600** — that's the color used for buttons, links, and accents. The other shades are used for backgrounds, hover states, and disabled elements.

| Shade | Usage |
|-------|-------|
| 50 | Background highlights, hover states |
| 100 | Light backgrounds |
| 200 | Borders, dividers |
| 300 | Disabled states |
| 400 | Placeholder text |
| 500 | Secondary text, icons |
| 600 | **Primary brand color** — buttons, links, accents |
| 700 | Hover states for buttons |
| 800 | Active/pressed states |
| 900 | Dark text, headings |

### Updating gradient colors to match

The CTA section gradient is set in `hugo.toml`. Update it to use your brand color as `gradient_from`:

```toml
[params.cta]
  gradient_from = "#2563eb"   # Your primary-600 brand color
  gradient_to = "#9333ea"     # Your secondary-600 brand color
  gradient_angle = 30
```

The hero section gradient (in `content/_index.md`) typically uses lighter shades for a subtle effect:

```markdown
{{< hero
    gradient-from="#dbeafe"   # primary-100 (light tint)
    gradient-to="#f3e8ff"     # secondary-100 (light tint)
    gradient-angle="180"
>}}
```


**How the sections work:**

- **`hero`**: The top section. `headline` can include HTML like `<span class='text-primary-600'>highlighted</span>`. The gradient uses lighter shades (100–200) for a subtle background.
- **`client-logos`**: Reads `client_logos` from the page's front matter and renders a scrolling logo bar. The data must be in front matter, not inline.
- **`features-section` + `feature`**: `features-section` is a wrapper — it must have a closing `{{< /features-section >}}` tag. Individual `feature` shortcodes go inside it. Alternate `imagePosition` between `"right"` and `"left"` for visual variety. Use distinct badge colors for each feature.
- **`testimonials`**: Reads `testimonials` from the page's front matter. Data is in front matter, not the shortcode body.
- **`cta`**: Takes no parameters — reads all configuration from `[params.cta]` in `hugo.toml`. This keeps CTA content consistent across all pages.

### Feature Pages

Create a directory at `content/features/` and add one file per feature. The front matter structure drives the feature layout — the `features` array populates a visual feature list, and `demo` adds an illustration.

Create `content/features/your-feature.md`:

```markdown
---
title: "Feature Name"
description: "Brief description of this feature and its value."
layout: "feature"
badge: "Category"
badgeColor: "#2563eb"
features:
  - title: "Sub-feature One"
    description: "Detailed description of this sub-feature and how it helps users."
  - title: "Sub-feature Two"
    description: "Detailed description of this sub-feature and how it helps users."
  - title: "Sub-feature Three"
    description: "Detailed description of this sub-feature and how it helps users."
  - title: "Sub-feature Four"
    description: "Detailed description of this sub-feature and how it helps users."
demo:
  description: "See this feature in action."
  image: "/images/feature-demo.svg"
---

## Detailed Description

Markdown content providing more detail about the feature. This appears below the structured feature list rendered by the layout.

### Key Benefits

- Benefit one with explanation
- Benefit two with explanation
- Benefit three with explanation

### How It Works

Step-by-step explanation of the feature.
```

Make sure the feature URLs in `hugo.toml` match the filenames. If the file is `content/features/analytics.md`, the URL is `/features/analytics/`.

### Blog

Create your first post at `content/blog/your-post-title.md`. Blog posts support categories, tags, and an auto-generated table of contents.

```markdown
---
title: "Blog Post Title"
date: 2024-01-15
author: "Author Name"
description: "Meta description for the blog post (150-160 characters)."
categories: ["Category"]
tags: ["tag1", "tag2", "tag3"]
featured_image: "/images/blog/post-image.jpg"
---

{{< toc >}}

## Introduction

Opening paragraph that hooks the reader.

## Main Section

Content with subheadings, lists, code blocks, images, etc.

### Subsection

More detailed content.

## Conclusion

Summary and call to action.
```

The `{{< toc >}}` shortcode auto-generates a table of contents from the page headings (H2 and H3 by default). The blog sidebar shows recent posts, categories, and tags automatically based on the front matter.

### Documentation

Documentation is structured as a section with a hub index page and child pages. The `weight` field controls sidebar ordering — lower weight = higher position.

Create the hub at `content/docs/_index.md`:

```markdown

title: Documentation

layout: single
---

# Documentation

Welcome to the documentation. This guide covers everything you need to get started.

## Quick Start

1. **[Getting Started](/docs/getting-started/)** - Installation and setup
2. **[Features](/docs/features/)** - Explore all features
3. **[Customization](/docs/customization/)** - Make it your own
```

```markdown
---
title: "Getting Started"

Step-by-step installation guide...
```

Each child page gets an entry in the doc sidebar automatically. Create as many child pages as needed, adjusting `weight` values to control the order.

## Step 8: Go Multilingual (Optional)

Saasify supports i18n out of the box. Here's a summary of what's involved.

### Language configuration in hugo.toml

Add language blocks to `hugo.toml`. Each language block overrides params for that language:

```toml
defaultContentLanguage = "en"

[languages]
  [languages.en]
    languageName = "English"
    weight = 1
    [languages.en.params]
      description = "Your English description"

  [languages.fr]
    languageName = "Français"
    weight = 2
    [languages.fr.params]
      description = "Votre description en français"
```

### Content directories

Place language-specific content in directories named by language code:

```
content/
├── en/
│   └── _index.md
├── fr/
│   └── _index.md
```

### Translation strings

Create `i18n/en.toml` and `i18n/fr.toml` for UI strings:

```toml
# i18n/en.toml
[readMore]
other = "Read More"

[getStarted]
other = "Get Started"
```

Hugo's multilingual system handles URL prefixes (`/fr/pricing`), language switchers, and separate menus per language automatically once the config is set up.

## Step 9: Preview and Launch

### Preview locally

```bash
npm start
```

Open http://localhost:1313 in your browser. The site rebuilds automatically when you save files. TailwindCSS compiles in watch mode alongside Hugo.

### Production build

```bash
npm run build
```

Output lands in the `public/` directory. This directory is what you deploy.

### Deployment options

**Netlify** — drag the `public/` folder into the Netlify UI, or connect the repo and set build command to `npm run build` and publish directory to `public`.

**Vercel** — same approach: set build command to `npm run build`, output directory to `public`.

**GitHub Pages** — push the `public/` directory to the `gh-pages` branch, or use a GitHub Action to run `npm run build` and deploy.

**Firebase Hosting** — `firebase init hosting` pointing at `public/`, then `firebase deploy`.

**Any static host** — upload the `public/` directory contents. The site is fully static with no server-side requirements.

### Updating the theme

```bash
git submodule update --remote themes/hugo-saasify-theme

# If package.json changed, re-copy and reinstall
cp themes/hugo-saasify-theme/package.json .
cp themes/hugo-saasify-theme/postcss.config.js .
cp themes/hugo-saasify-theme/tailwind.config.copy.js ./tailwind.config.js
npm install
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| CSS not loading | Ensure `writeStats = true` in hugo.toml `[build]` section |
| Theme not found | Check `theme = "hugo-saasify-theme"` in hugo.toml |
| Hugo Extended error | Download Hugo Extended edition from GitHub releases |
| Empty theme directory | Run `git submodule update --init --recursive` |
| npm install fails | Delete `node_modules/` and `package-lock.json`, then `npm install` |
| Port 1313 in use | Kill process: `lsof -ti:1313 \| xargs kill -9` then retry |
| Shortcode JSON errors | Check that all JSON uses double quotes, not single quotes |
| Testimonials not showing | Data must be in page front matter, not in the shortcode body |
| Client logos not showing | Data must be in page front matter, not in the shortcode body |
